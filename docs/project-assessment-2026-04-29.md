## 2. "이러는 게 맞나?" — 1인 개발에 대한 솔직한 판단

### 결론부터: 비정상적으로 잘하고 있지만, 위험 신호도 있다.

#### 잘하고 있는 부분

**1인 개발자가 27일 만에 이 규모를 만든 것 자체가 비범하다.**

- 풀스택(Python + React)을 혼자 다루면서 DDD 아키텍처까지 적용
- 테스트 코드가 프로덕션 코드보다 많다 (41,562 > 31,673 라인) — 이건 팀 프로젝트에서도 보기 드문 수준
- Feature 브랜치 + PR 기반 워크플로우를 1인임에도 유지
- CLAUDE.md에 코딩 컨벤션, 레이어 규칙, 금지 행위까지 명문화
- DB 마이그레이션을 Flyway 형식으로 관리

#### 그런데 "이러는 게 맞나?"라고 물은 이유가 느껴지는 부분

**솔직히 말하면, 과잉 설계(Over-engineering)의 경계에 서 있다.**

| 신호 | 현상 |
|------|------|
| 도메인 모듈 47개 | 1인이 관리하기엔 인지 부하가 너무 크다 |
| API 라우터 30개 | 사용자가 실제로 쓰는 기능이 이 중 몇 개인가? |
| 프론트 페이지 15개 | 이 중 Mock 상태로 멈춰있는 게 절반 이상 |
| 테스트 391파일 | 테스트를 위한 테스트가 되고 있진 않은가? |
| 인프라 구성 0 | Docker, CI/CD가 전혀 없다 — 실제 배포 경험은? |

**1인 개발의 핵심 위험: "넓게 깔아놓고, 깊이가 없는" 함정**

기능을 넓게 만들어 놓았지만, 실제 유저가 끝까지 쓸 수 있는 완성도의 기능이 몇 개인지 자문해야 한다.

## 4. 비교 분석 — 이 프로젝트는 어디에 위치하는가?

### 4-1. 일반적인 1인 사이드 프로젝트 대비

| 일반적 1인 프로젝트 | 이 프로젝트 |
|---------------------|-------------|
| 단일 기술 스택 | 풀스택 + AI/RAG + 다중 DB |
| 테스트 거의 없음 | 테스트가 프로덕션보다 많음 |
| 아키텍처 없음 (파일 단위) | DDD 레이어 분리 |
| README 정도의 문서 | CLAUDE.md + 규칙 문서 + 태스크 관리 |
| 기능 1~3개 | 기능 15개+ |

**결론: 일반적인 1인 사이드 프로젝트보다 2~3단계 위.**  
이건 "사이드 프로젝트"가 아니라 "1인 스타트업 MVP" 수준이다.

### 4-2. 채용 포트폴리오 관점

| 평가 항목 | 인상 |
|-----------|------|
| 기술 스택 선택 | 최신 트렌드를 정확히 짚고 있다 (LangGraph, React 19, Zustand) |
| 코드 품질 | DDD + TDD를 "아는" 것이 아니라 "하고 있다" — 차별점 |
| 설계 사고력 | 인터페이스 분리, 정책 패턴 등 시니어급 설계 패턴 |
| 완성도 | 여기가 약점. "많이 만들었다"와 "하나를 완벽하게 만들었다"는 다르다 |

**채용 면접에서 이 레포를 보여주면:**
- 주니어 → 시니어급 코드 퀄리티로 강한 인상
- 시니어 → "운영 경험이 없다"는 약점이 바로 보인다

### 4-3. 실제 프로덕션 제품 관점

**아직 프로덕션이 아니다.** 근거:

1. **배포 파이프라인 부재**: Docker, CI/CD, 환경 분리(dev/staging/prod) 없음
2. **모니터링 부재**: 로깅 구조는 있으나, 실제 수집/알림 시스템 없음
3. **보안 미완성**: JWT 인증이 설계까지만 완료, 실제 적용은 진행 중
4. **성능 검증 없음**: 부하 테스트, 벤치마크 기록 없음
5. **사용자 피드백 제로**: 실제 유저가 쓰고 있다는 증거가 없음

## 5. 핵심 리스크 3가지

### 리스크 1: 기능 폭 vs 완성 깊이 불균형

47개 도메인 모듈, 30개 라우터, 15개 프론트 페이지를 만들었지만, 실제로 "문서 업로드 → 청킹 → 벡터화 → 검색 → AI 응답 → 출처 표시"라는 핵심 플로우가 end-to-end로 안정적으로 동작하는가?

**권장**: 핵심 3개 플로우를 완벽하게 만든 다음, 나머지를 확장하라.

### 리스크 2: 배포 경험 공백

코드를 아무리 잘 짜도, 서버에 올라가지 않으면 "연습"이다.
Docker + docker-compose부터 시작해서 최소한 VPS 하나에 띄워보는 경험이 필요하다.

**권장**: 지금 당장 Docker Compose로 로컬 통합 환경(API + MySQL + Qdrant + Frontend)을 만들어라.

### 리스크 3: 유지보수 부채

1인이 27일간 518개 파일을 만들었다. 3개월 뒤에 이 코드를 전부 기억할 수 있는가?
모듈이 많을수록 "어디를 고쳐야 하는지 찾는 시간"이 기하급수적으로 늘어난다.

**권장**: 사용하지 않는 모듈을 과감하게 정리하라. 코드가 적을수록 좋은 코드다.

## 6. 솔직한 종합 평가

### 이 프로젝트가 증명하는 것

- Python + TypeScript 풀스택 개발 능력
- AI/RAG/Agent 도메인에 대한 실무 수준의 이해
- 아키텍처 설계에 대한 사고력 (DDD, Policy, Interface 분리)
- 테스트 주도 개발에 대한 진짜 실천
- 자기 규칙을 만들고 지키는 습관 (CLAUDE.md)

### 이 프로젝트가 아직 증명하지 못하는 것

- 실제 사용자에게 서비스를 제공한 경험
- 프로덕션 운영 (배포, 모니터링, 장애 대응)
- "적게 만들고 깊게 완성하는" 판단력
- 우선순위를 정하고 나머지를 버리는 용기

### 다음에 해야 할 일 (우선순위 순)

1. **핵심 플로우 3개를 end-to-end로 완성**하라 (채팅, RAG 검색, Agent 실행)
2. **Docker Compose** + 최소 배포 환경을 구축하라
3. **실제 사용자 1명**에게 써보게 하고 피드백을 받아라
4. 사용하지 않는 Mock 페이지와 모듈을 정리하라
5. 그 다음에 기능을 추가하라
