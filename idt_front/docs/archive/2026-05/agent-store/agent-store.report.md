## 6. 배운 점 및 회고

### 6.1 잘한 점 (계속 유지)

1. **설계 문서의 상세함**
   - Design 문서에 7개 섹션 (타입, 엔드포인트, 서비스, 쿼리 키, 훅, 컴포넌트, 페이지)으로 명확한 구조
   - 구현 순서(Step 1~5)가 명확하여 개발 효율성 증대
   - **효과**: 초기 설계 매칭율 88% → 개선 후 96% (반복 1회로 완료)

2. **TDD 접근법**
   - 훅과 컴포넌트에 대한 테스트를 구현과 함께 작성
   - MSW를 활용한 API 모킹으로 백엔드 의존성 제거
   - **효과**: 버그 조기 발견, 코드 품질 향상

3. **모듈화된 API 서비스 계층**
   - `agentStoreService`에 모든 API 호출 집중
   - `authApiClient` 활용으로 인증 처리 자동화
   - **효과**: 엔드포인트 변경 시 서비스 계층만 수정

4. **TanStack Query 활용**
   - 8개 훅(useAgentList, useAgentDetail, useMyAgents, useSubscribeAgent, useUnsubscribeAgent, useUpdateSubscription, useForkAgent, useForkStats)으로 깔끔한 데이터 페칭
   - 뮤테이션 성공 시 자동 캐시 무효화
   - **효과**: 로딩 상태, 에러, 캐싱 자동 관리

### 6.2 개선 필요 사항 (문제점)

1. **에러 처리의 일관성 부족**
   - 초기 설계에서 toast 알림을 명시했으나, 실제 구현 시 인라인 에러 텍스트 사용
   - **이유**: Toast 컴포넌트 라이브러리 통일 필요 (현재 프로젝트에서 표준화되지 않음)
   - **개선안**: 전사 Toast 컴포넌트 표준화 후 적용

2. **백엔드 의존성 관리**
   - PublishAgentModal의 department 드롭다운이 백엔드 `/api/v1/departments` API 미지원으로 구현 불완
   - **이유**: 백엔드 개발 스케줄 지연
   - **해결책**: 사용자 입력 텍스트로 폴백 (향후 API 추가 시 업그레이드)

3. **테스트 커버리지 (P2 항목)**
   - AgentDetailModal.test.tsx 미작성 (P2 비블로킹)
   - **이유**: 시간 제약으로 우선순위 P1 항목 먼저 완료
   - **개선안**: 다음 사이클에서 추가 작성 (2시간 소요)

### 6.3 다음 사이클에 적용할 점 (Try)

1. **전사 Toast 컴포넌트 통일**
   - 프로젝트 전체에서 일관된 toast 알림 사용
   - shadcn/ui의 sonner 또는 react-hot-toast 표준화

2. **백엔드 API 사전 정의**
   - 프론트엔드 설계 시 필요한 모든 엔드포인트를 백엔드와 사전 논의
   - API 우선 설계(API-First Design) 도입

3. **P2 항목도 TDD 커버**
   - 향후 기능에서 P2 항목의 테스트도 함께 작성
   - 목표 테스트 커버리지: 75% → 85% 이상

4. **설계-구현 가이드라인 문서화**
   - 이번 성공 사례를 프로젝트 가이드에 포함
   - 다른 기능 구현 시 참조 가능하도록 패턴화
