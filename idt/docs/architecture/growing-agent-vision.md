# 성장형 에이전트(Growing Agent) — 궁극적 목표 아키텍처 비전

> 작성일: 2026-07-18
> 성격: 방향성 문서 (개별 기능의 Plan/Design은 `docs/01-plan`, `docs/02-design`에 별도 작성)
> 관련 자산: LLM Wiki(V036), Retrieval Observability(V046), search_history(V049), admin dashboard, agent-memory(Plan 작성됨)

---

## 1. 비전 선언

> **"매번 설명해야 하는 AI"에서, 사용할수록 조직과 사용자를 이해하는 AI로.**

현재 시스템은 벡터 DB 기반 RAG로 "문서를 아는" 단계까지 왔다. 그러나 무상태(stateless)라서
사용자가 누구인지, 조직에서 용어가 어떤 의미인지, 과거에 어떤 답이 맞았는지는 매 질문마다 잊는다.
엔터프라이즈 환경에서 일반 사용자는 이런 문맥을 스스로 주입하지 못한다 — **문맥 주입 부담을
사용자에게서 시스템으로 옮기는 것**이 이 비전의 전부다.

여기서 "성장"은 모델 파인튜닝이 아니다. 금융/정책 도메인의 보수적 동작 요구(비용·재현성·컴플라이언스)
때문에 모델 가중치는 고정하고, **모델 바깥의 세 겹이 데이터로 성장한다**:

| 성장 축 | 실체 | 예시 |
|---------|------|------|
| ① 메모리 (문맥) | 사용자/조직 문맥의 구조화 저장 → 프롬프트 자동 주입 | "심사팀 소속, 근거 조문 인용 선호" |
| ② 검색 품질 (피드백) | 평가 데이터 기반 랭킹·질의 개선 | 👍 받은 답의 (질의, 청크) 쌍 부스팅 |
| ③ 지식 큐레이션 (위키) | 검증된 지식의 승격·정제 | 高평가 답변 → 승인 → WikiArticle |

## 2. 목표 아키텍처 전체 그림

```
                        ┌────────────────────────────────────────────┐
                        │              사용자 질문                     │
                        └────────────────┬───────────────────────────┘
                                         ▼
 [주입 계층]      ┌──────────────────────────────────────────────────┐
 Tier 0 상주      │ 시스템 프롬프트 = 기본 지침                          │
 (하드캡)         │   + 사용자 메모리(profile/preference/용어)  ◀── agent_memory (MySQL)
                 │   + org 메모리 인덱스(한 줄 목록)                     │
                 └──────────────────────┬───────────────────────────┘
                                        ▼
 [추론/검색]      ┌──────────────────────────────────────────────────┐
 온디맨드         │ LangGraph Agent                                   │
                 │   ├─ Wiki-First Search (approved 위키 우선)  ◀── wiki_article (기존 V036)
                 │   ├─ Routed Retrieval (요약 계층 3단계)              │
                 │   └─ read_memory(slug) 도구 (org 메모리 본문)        │
                 └──────────────────────┬───────────────────────────┘
                                        ▼
                 ┌──────────────────────────────────────────────────┐
                 │ 답변  + retrieval trace (run_tracker, 기존 V046)    │
                 └──────┬───────────────────────────────────────────┘
                        ▼
 [피드백 수집]    ┌──────────────────────────────────────────────────┐
                 │ 👍/👎 + 사유  ──▶ feedback 테이블 (run_id FK)        │
                 │ 암묵 신호(재질문·수정 요청)                           │
                 └──────┬───────────────────────────────────────────┘
                        ▼ (비동기 백그라운드)
 [성장 계층]      ┌──────────────────────────────────────────────────┐
                 │ 추출기: 대화에서 기억할 사실 후보 추출 (LLM)            │
                 │ 정합기: 기존 메모리와 비교 → 신규/갱신/모순/무시          │
                 │ 사서 잡(reflection): 저수준 기록 → 고수준 통찰 압축      │
                 │ 승격기: 高평가 답변 → WikiArticle draft (기존 distill 재사용)
                 └──────┬───────────────────────────────────────────┘
                        ▼
 [거버넌스]       ┌──────────────────────────────────────────────────┐
                 │ user scope: 자동 active (본인 열람·삭제 가능 — 투명성)  │
                 │ org scope:  pending → 관리자 승인 → active            │
                 │             ◀── admin dashboard 승인 큐 (기존 화면 확장)│
                 │ 평가 게이트: 골든 QA 셋 회귀 (LangSmith) — 성장 ≠ 표류   │
                 └──────────────────────────────────────────────────┘
```

## 3. 핵심 설계 원칙 (불변)

1. **모델은 고정, 데이터가 성장한다** — 파인튜닝 없이 메모리·랭킹·위키의 3축으로 성장.
2. **사람 승인 게이트** — 조직 공유 지식(org 메모리, 위키)은 반드시 `pending → 관리자 승인 → active`.
   한 사용자의 오답이 다수결로 전사 지식이 되는 피드백 오염을 구조적으로 차단한다.
   (기존 LLM Wiki의 draft→approve 게이트와 동일 원칙 — 이미 검증된 패턴.)
3. **출처 불변식** — 모든 성장 산출물은 원본으로 역추적 가능해야 한다.
   메모리는 `source_run_id`, 위키는 `source_refs`. "왜 이렇게 답했나"를 감사할 수 있는 구조 자체가
   금융 도메인의 셀링 포인트다.
4. **컨텍스트는 캡이 있다** — 메모리 시스템의 성패는 무엇을 기억하느냐가 아니라
   **매 순간 무엇을 컨텍스트에 안 넣느냐**로 갈린다. Tier 0 상주(하드캡 ≈800토큰) +
   인덱스 상주/본문 온디맨드(위키 패턴) + reflection 압축으로 무한 누적을 구조적으로 차단.
5. **성장은 측정 없이는 표류다** — 메모리 주입·부스팅·위키 승격이 바뀔 때마다
   골든 QA 셋 회귀 평가를 통과해야 반영된다.
6. **투명성 = 신뢰** — 사용자는 "AI가 나에 대해 아는 것"을 언제든 열람·수정·삭제할 수 있다.
7. **격리** — 성장 계층(추출·정합·승격)은 전부 비동기 백그라운드. 성장 기능의 장애가
   답변 경로의 장애로 전파되지 않는다.

## 4. 현재 자산 → 목표 구성 요소 매핑

이 비전은 제로 베이스가 아니다. 배관의 상당 부분이 이미 있다.

| 목표 구성 요소 | 현재 자산 | 상태 |
|----------------|-----------|------|
| retrieval trace (피드백의 원인 분석 기반) | `get_run_tracker()` 관측성 (V046), `ai_run`/`ai_llm_call` | ✅ 완료 |
| 검색 이력 | `search_history` (V049, kb_id 격리) | ✅ 완료 |
| 지식 큐레이션 + 승인 게이트 | **LLM Wiki** (V036): distill → draft → 관리자 승인 → wiki-first search | ✅ 완료 |
| 관리자 관제 화면 (승인 큐 수용처) | admin dashboard, `/admin/wiki` | ✅ 완료 |
| 사용자 메모리 저장·주입 (Tier 0) | **agent-memory Phase 1** — Plan 작성됨 (V050 예정) | 🔄 진행 |
| 답변 피드백 수집 (👍/👎 + run_id 연결) | 없음 — feedback-loop feature로 기획 예정 | ⏳ 미착수 |
| 자동 추출·정합기 | 없음 (agent-memory Phase 2) | ⏳ 미착수 |
| org 메모리·승인 큐·reflection 사서 잡 | 없음 (agent-memory Phase 3) | ⏳ 미착수 |
| 高평가 답변 → 위키 승격 | 없음 — 기존 distill 파이프라인 재사용 예정 | ⏳ 미착수 |
| 피드백 기반 검색 부스팅/리랭킹 | 없음 — 피드백 데이터 축적 후 | ⏳ 후순위 |
| 골든 QA 셋 평가 게이트 | LangSmith 연동은 존재, dataset/evaluator 미구축 | ⏳ 미착수 |

**특기**: LLM Wiki가 이미 "자동 생성 → 승인 게이트 → 우선 검색"의 전체 사이클을 구현했다.
성장형 에이전트의 지식 큐레이션 축은 신규 개발이 아니라 **위키의 입력 소스를 확장**하는 문제다
(현재: 문서 청크 distill → 추가: 高평가 답변 승격).

## 5. 로드맵

각 단계는 독립적으로 가치를 내고, 다음 단계의 기반이 된다. 순서의 근거는
**"주입 경로 검증 → 신호 수집 → 자동화 → 공유화"** — 뒤 단계는 앞 단계의 데이터 없이는 무의미하다.

### Phase 1 — 수동 메모리 + 주입 (agent-memory, Plan 완료)
- `agent_memory` 테이블(V050, tier/scope/status 확장 컬럼 선반영)
- 사용자 본인 CRUD + SettingsPage 관리 UI + General Chat 프롬프트 주입(하드캡)
- **검증 목표**: 메모리 주입이 답변 품질을 올리는가 (추출 난이도 0에서 순수 검증)

### Phase 2 — 피드백 수집 (feedback-loop)
- 답변 단위 👍/👎 + 사유 → `feedback` 테이블 (run_id FK로 retrieval trace 연결)
- admin dashboard에 피드백 통계·低평가 답변 드릴다운
- **검증 목표**: 어디서 틀리는지가 데이터로 보이는가

### Phase 3 — 자동 추출·정합 (agent-memory Phase 2)
- 대화 종료 시 비동기 추출기(저비용 LLM, structured output) → 정합기(신규/갱신/모순/무시 판정)
- user scope 자동 active, 사용자 열람·삭제 UI는 Phase 1 재사용
- **검증 목표**: 자동 축적된 메모리의 정밀도 (오염 없이 쌓이는가)

### Phase 4 — 조직 지식화 (agent-memory Phase 3 + 위키 승격)
- org 메모리(용어 사전·교정 규칙): pending → admin 승인 큐 → active
- 高평가 답변 → 기존 wiki distill 파이프라인으로 draft 승격
- 인덱스 상주 + `read_memory` 온디맨드 도구 (위키 패턴)
- reflection 사서 잡: 저수준 기록 압축·중복 병합·만료
- **검증 목표**: 한 사용자의 학습이 조직 전체의 답변 품질로 전이되는가

### Phase 5 — 랭킹 성장 + 평가 게이트
- 골든 QA 셋 + LangSmith evaluator — 이후 모든 성장 반영의 회귀 게이트
- 피드백 (질의, 청크, 채택) 데이터 기반 검색 부스팅 → (데이터 충분 시) 리랭커
- **검증 목표**: 성장 루프가 지표를 실제로 올리는가 (표류 방지)

## 6. 이 문서가 아닌 것

- 개별 기능의 스펙이 아니다 — 각 Phase는 착수 시 `/pdca plan → design`으로 별도 구체화한다.
- 확정 일정이 아니다 — 각 Phase의 검증 목표가 실패하면 다음 단계로 가지 않고 방향을 수정한다.
- 아키텍처 변경 승인이 아니다 — Thin DDD 레이어 규칙, 대화 원문의 vector DB 저장 금지 등
  기존 불변 규칙(`idt/CLAUDE.md`)은 모든 Phase에서 유지된다.
