## 1. 판정 요약

> **정합. LLM Wiki는 비전의 "지식 큐레이션 축"을 선행 구현한 자산이며, 방향 충돌은 없다.**
> 비전의 7개 설계 원칙 중 4개를 이미 구현으로 충족하고, 나머지 3개는 위키를 고치는 게 아니라
> **위키에 연결선을 추가**(피드백→confidence, 환류 생성 경로, 평가 게이트)하면 충족된다.
> 조정이 필요한 지점은 단 하나 — **스코프 모델**(위키의 agent 단위 vs 비전의 org 단위)이다.

| 판정 항목 | 결과 |
|-----------|------|
| 방향 충돌 여부 | **없음** — 승인 게이트·출처 불변식 등 핵심 원칙이 이미 위키에 코드로 존재 |
| 재사용 가능성 | **높음** — 환류 생성용 enum(`CONVERSATION`)·confidence 갱신 함수(`clamp_confidence`)가 예약돼 있어 확장 자리까지 마련됨 |
| 구조 조정 필요 | **1건** — 스코프 모델 (agent 단위 → org 공유 요구와의 정렬, §5-1) |
| 폐기·재작성 대상 | **없음** |

한 장으로 요약하면 — **위키는 아래 비전 순환 고리의 오른쪽 절반(지식이 쌓이고 검증되어 검색에 쓰이는 길)을
이미 완성했고, 왼쪽 절반(사용자 평가가 그 지식을 만들어내는 길)이 비어 있다**:

```
                    ┌───────────── 성장형 에이전트 순환 고리 ─────────────┐
                    │                                                    │
   [비어 있음 ⏳]                                        [이미 완성 ✅]
   사용자 평가(👍/👎)                                     검증된 지식 저장소
        │                                                      ▲
        │  피드백 수집 ──▶ 승격 후보 ──▶ 관리자 승인 ──▶ WikiArticle(approved)
        │   (Phase 2)      (Phase 4)      ✅ 있음              │
        │                                                      │ ✅ 있음
        ▼                                                      ▼
   답변 품질 개선  ◀──────────────────────────  위키 우선 검색(wiki-first)
                    │                                                    │
                    └────────────────────────────────────────────────────┘
```

## 2. LLM Wiki 현재 구현 현황 (As-Is 실측 스냅샷)

### 2-1. 현재 아키텍처 전체 그림

지식이 들어오는 길(왼쪽 위), 사람이 검증하는 길(가운데), 에이전트가 쓰는 길(아래)의 3단 구조다.

```
┌──────────────────────────── ① 생성 (지식이 들어오는 길) ────────────────────────────┐
│                                                                                     │
│   관리자                                                                             │
│     │  POST /wiki/distill (수동 트리거, admin 권한)                                    │
│     ▼                                                                               │
│   DistillToWikiUseCase                                                              │
│     │ 1. ES에서 원본 청크 조회 (최대 200건) ──── Elasticsearch (원본 문서 청크)          │
│     │ 2. 문서(source) 단위로 그룹핑                                                    │
│     │ 3. LLM 정제 ("원문 사실만, 추측 금지") ─── ChatOpenAI (gpt-4o-mini, temp=0)      │
│     │ 4. 불변식 검증: source_refs ≥ 1  ◀━━ 출처 없으면 생성 자체가 불가                  │
│     ▼                                                                               │
│   WikiArticle (status=draft)   ※ 이 시점엔 검색에 절대 노출되지 않음                     │
│                                                                                     │
└──────────────────────────────────────┬──────────────────────────────────────────────┘
                                       ▼
┌──────────────────────────── ② 거버넌스 (사람이 검증하는 길) ─────────────────────────┐
│                                                                                     │
│   /admin/wiki 관리 화면 (WikiPage)                                                   │
│                                                                                     │
│      draft ──approve(reviewer_id 기록)──▶ approved ──deprecate──▶ deprecated         │
│        │                                     ▲                        │             │
│        └───────reject──▶ deprecated          └───────restore──────────┘             │
│                                                                                     │
│   검색 노출 조건 = status==approved AND 미만료(valid_until)  → is_searchable()         │
│                                                                                     │
└──────────────────────────────────────┬──────────────────────────────────────────────┘
                                       ▼
┌──────────────────────────── ③ 저장 & 검색 (에이전트가 쓰는 길) ──────────────────────┐
│                                                                                     │
│   저장 이원화:  MySQL wiki_article ◀━━ 상태의 Source of Truth                         │
│                Qdrant wiki_knowledge ◀━ 본문 임베딩 (유사도 검색용)                     │
│                                                                                     │
│   사용자 질문                                                                         │
│     ▼                                                                               │
│   LangGraph Agent → InternalDocumentSearchTool                                      │
│     │   ※ RAG 도구별 use_wiki_first=true 옵트인 시에만 (전역 아님)                      │
│     ▼                                                                               │
│   RunScopedWikiSearch → WikiFirstSearchUseCase                                      │
│     │ 1. Qdrant 유사도 검색 (agent_id 필터)                                            │
│     │ 2. MySQL 하이드레이션 → is_searchable() 최종 필터 (Qdrant 메타가 낡아도 안전)      │
│     │ 3. 결과 부족분 → 기존 HybridSearch 폴백 (위키 우선 병합)                           │
│     ▼                                                                               │
│   답변  (위키 유래 결과는 source="wiki" 마킹 — 식별 가능)                                │
│                                                                                     │
└─────────────────────────────────────────────────────────────────────────────────────┘
```

| 영역 | 상태 | 근거 |
|------|------|------|
| 도메인 모델 (엔티티·정책·상태 전이) | ✅ 완성 | `src/domain/wiki/entity.py`, `policies.py` |
| 자동 생성 (distill) | ✅ 완성 (관리자 수동 트리거) | `application/wiki/distill_use_case.py` |
| 승인 워크플로 + 관리 UI | ✅ 완성 | `review_use_case.py`, `idt_front/src/pages/WikiPage/` |
| 위키 우선 검색 + 폴백 | ✅ 완성, 런타임 배선됨 | `wiki_first_search_use_case.py`, `main.py:2247` |
| 저장 이원화 (MySQL SoT + Qdrant) | ✅ 완성 | `infrastructure/wiki/wiki_repository.py` |
| 테스트 | ✅ 7개 영역 | `tests/**/wiki/` |

### 2-2. 무엇이 비어 있나 (가이드 §10 + 실코드 확인)

| 한계 | 내용 | 비전에서의 의미 |
|------|------|----------------|
| 환류 생성 미구현 | `CONVERSATION`/`WEBSEARCH` source_type은 **enum 예약만** (`entity.py:16-17`) | 비전 Phase 4 "高평가 답변 승격"의 예정된 자리 |
| confidence 갱신 미구현 | `WikiPolicy.clamp_confidence`만 준비, 갱신 로직 없음 | 비전 Phase 2 피드백 연동의 예정된 자리 |
| ES(BM25) 위키 색인 없음 | 위키 검색은 Qdrant 벡터만 | 품질 최적화 후속 (방향과 무관) |
| human 신규 작성 API 없음 | 편집만 가능, 신규는 distill 경로뿐 | org 용어 사전(수동 등록) 요구 시 필요 |
| 만료 자동 정리 없음 | 검색 제외만 되고 배치 정리 없음 | 비전 원칙 4(캡·정리) 관점의 소과제 |
| **distill 중복 검사 없음** | 같은 컬렉션에 distill 재실행 시 동일 문서에서 draft가 또 생성됨 (`distill_use_case.py`에 기존 항목 조회 로직 부재 — 2026-07-18 실코드 확인). 현재는 draft 단계 사람 검수가 유일한 방어선이며, approved 항목 간 중복을 막는 장치도 없음 | 코퍼스 위생 리스크 — 사서(reflection) 잡(Phase 4)의 병합 대상이나, distill 자체의 중복 검사는 로드맵과 무관하게 지금도 고칠 가치 있는 소과제 |

## 3. As-Is 위에 To-Be를 겹쳐 보기 — 무엇이 있고 무엇을 추가하나

비전 목표 아키텍처에 현재 위키 구현을 겹친 그림이다.
**실선 박스(✅) = 이미 프로덕션에 있는 것, 점선 박스(⏳) = 비전 로드맵에서 추가할 것.**
점선이 전부 "위키 수정"이 아니라 "위키로 들어오는/나가는 연결선"임에 주목.

읽는 법: 오른쪽 아래 순환(승인된 지식 → 검색 → 답변)은 **오늘 이미 돌고 있다.**
비어 있는 것은 답변에서 지식으로 돌아오는 **왼쪽 화살표들**(피드백 → 환류 → 승격)이고,
그 화살표가 꽂힐 자리(enum·함수)는 위키에 예약돼 있다.

## 4. 비전 설계 원칙 7개 대비 정합성

| # | 비전 원칙 | 위키 현재 상태 | 판정 |
|---|-----------|---------------|------|
| 1 | 모델은 고정, 데이터가 성장 | distill로 지식 데이터가 성장, 모델 무관 | ✅ 충족 |
| 2 | 사람 승인 게이트 (pending→승인→active) | `draft→approve` 게이트가 정확히 이 원칙의 선행 구현. 자동 생성물은 무조건 draft, 승인 전 검색 비노출 | ✅ 충족 — **비전이 위키의 검증된 패턴을 역으로 채택한 관계** |
| 3 | 출처 불변식 (역추적 가능) | `source_refs ≥ 1` 없으면 생성 자체가 불가 (`validate_for_creation`) + reviewer_id/editor_id/version 감사 추적 | ✅ 충족 |
| 4 | 컨텍스트 하드캡 + 온디맨드 | 위키는 프롬프트 상주가 아니라 **검색 경유(온디맨드)** — 상주 부담 0. 본문 8,000자 상한 존재 | ✅ 충족 (구조적으로) |
| 5 | 성장은 측정 없이는 표류 | confidence가 score로 쓰이지만 **갱신 신호가 없음**(고정 0.5). 골든 QA 셋·회귀 게이트 부재 | ❌ 갭 — Phase 2·5에서 연결 |
| 6 | 투명성 (열람·통제) | 관리자에겐 완전 투명(/admin/wiki). 답변에서 `source="wiki"`로 유래 식별 가능. 일반 사용자용 "이 답의 근거 위키 보기" UI는 없음 | 🔶 부분 — 관리자 투명성만 |
| 7 | 격리 (성장 계층은 비동기) | distill이 **동기 API**(관리자 대기) — 수동 트리거라 현재는 문제없으나, 환류 자동화 시 비동기 전환 필요 | 🔶 부분 — Phase 4 전환 조건 |

**요약**: 원칙 2·3(거버넌스·출처)이라는 가장 어려운 부분이 이미 코드로 검증돼 있다.
갭인 5·6·7은 위키 구조의 결함이 아니라 **아직 안 만든 연결선**이며, 전부 비전 로드맵의 해당 Phase에 예정돼 있다.

### 로드맵 단계별 위키의 역할

| 비전 Phase | 위키의 역할 | 필요한 작업 | 재사용되는 기존 코드 |
|------------|-------------|------------|---------------------|
| 1. 수동 메모리 (진행 중) | 없음 — 메모리(문맥)와 위키(지식)는 별개 축 (§5-2) | 없음 | — |
| 2. 피드백 수집 | 위키 유래 답변(`source="wiki"`)에 대한 👍/👎를 confidence 환류 | feedback→confidence 갱신 유스케이스 1개 | `clamp_confidence`(준비됨), run_tracker 연결 |
| 4. 조직 지식화 | **高평가 답변 → 위키 draft 승격** — 비전의 핵심 성장 루프 | `source_type=CONVERSATION` 생성 유스케이스 (승인 흐름·UI는 무변경 재사용) | distill 파이프라인 골격, 승인 워크플로 전체, /admin/wiki UI |
| 5. 평가 게이트 | 위키 승격·confidence 변경도 골든셋 회귀 대상에 포함 | LangSmith dataset/evaluator | wiki-first search의 source 마킹(효과 측정용) |

## 5. 방향 차이 및 조정 필요 지점

### 5-1. 스코프 모델 — 유일한 실질 조정 대상

위키 지식은 현재 **에이전트 단위로 격리**돼 있다. 에이전트 A가 승인받은 지식을 에이전트 B는 못 본다.
비전의 org 공유 지식(전사 용어 사전)과 스코프가 어긋나는 유일한 지점이다:

```
        현재 (agent 스코프)                     비전이 요구하는 것 (org 공유)
  ┌─────────────┐ ┌─────────────┐          ┌───────────────────────────┐
  │  Agent A    │ │  Agent B    │          │        조직 공유 지식        │
  │             │ │             │          │   (용어 사전·교정 규칙)       │
  │  위키 A     │ │  위키 B      │          └─────────────┬─────────────┘
  │  (A만 검색)  │ │  (B만 검색)  │                 ┌───────┼───────┐
  └─────────────┘ └─────────────┘                 ▼       ▼       ▼
     서로 못 봄 ✗                              Agent A  Agent B  Agent C
                                              (모두 검색 가능해야 함)
```

조정 방향은 Phase 4 Design에서 확정하되, 이 프로젝트의 확립된 선호
(기존 필드 갈아끼우기 금지, **독립 opt-in 신규 필드 + 기존 동작 보존**)에 따라 두 후보를 둔다:

- 후보 ①: `wiki_article`에 `scope('agent'|'org')` 컬럼 추가 — 기본 `agent`(기존 동작 불변), `org`는 agent_id 필터를 우회해 전 에이전트 검색에 포함
- 후보 ②: org 지식은 위키가 아니라 `agent_memory`(Tier 1, org scope)가 담당하고 위키는 문서 지식 전담 — 역할 분리 유지

어느 쪽이든 **기존 agent 스코프 동작은 기준선으로 보존**한다.

### 5-2. 위키 ≠ 메모리 — 대체 관계가 아니라 병렬 축

혼동하기 쉬운 지점: 위키가 있는데 agent-memory가 왜 필요한가?
둘은 답변 생성의 **다른 단계**에 꽂히는 다른 종류의 지식이다:

```
                         사용자 질문
                             │
        ┌────────────────────▼─────────────────────┐
        │ 1단계: 프롬프트 조립                        │
        │   agent-memory가 여기 꽂힘                  │
        │   "누가, 어떻게 원하는가"                    │      항상 주입 (상주)
        │   예) 심사팀 소속 / 조문 인용 선호            │      토큰 하드캡
        └────────────────────┬─────────────────────┘
                             ▼
        ┌──────────────────────────────────────────┐
        │ 2단계: 지식 검색                            │
        │   LLM Wiki가 여기 꽂힘                      │
        │   "무엇이 사실인가"                          │      히트될 때만 사용
        │   예) 동일인 여신한도 규정의 정제된 요약        │      (온디맨드)
        └────────────────────┬─────────────────────┘
                             ▼
                           답변
```

| | LLM Wiki | agent-memory |
|---|----------|--------------|
| 담는 것 | **무엇이 사실인가** (문서·답변에서 정제된 지식) | **누가·어떻게 원하는가** (사용자 프로필·선호·용어 문맥) |
| 소비 방식 | 검색 시 히트되면 사용 (온디맨드) | 매 요청 프롬프트 상주 주입 (Tier 0) |
| 스코프 | 에이전트 (→ org 확장 검토) | 사용자 (→ org 확장 예정) |
| 근거 축 | 비전 3축 중 ③ 지식 큐레이션 | 비전 3축 중 ① 메모리 |

### 5-3. 트리거 방향 — 풀(pull)에서 푸시(push)로의 "추가"

```
   현재: 풀(pull) 모델                        비전 추가분: 푸시(push) 모델
   ─────────────────────                     ─────────────────────────
   관리자가 생각날 때                          사용자 평가가 쌓이면
   수동으로 distill 트리거                     시스템이 승격 후보를 자동 생성
        │                                         │
        ▼                                         ▼
   문서 → 지식  (초기 적재용, 계속 유효)        답변 → 지식  (운영 중 성장 담당)
        │                                         │
        └──────────── 둘 다 draft → 관리자 승인 게이트 통과 (동일) ────────────┘
```

방향 전환이 아니라 생성 경로의 추가다. 단, 푸시 자동화 시점에 승격 작업은
비동기 잡으로 전환한다(비전 원칙 7 — 성장 계층 장애가 답변 경로로 전파 금지).

---

## 6. 결론

1. **방향은 맞다.** LLM Wiki는 비전과 독립적으로 만들어졌지만, 비전의 가장 어려운 요구
   (자동 생성물의 거버넌스·출처 추적)를 이미 프로덕션 코드로 충족한다. 비전 문서의 승인 게이트
   원칙은 사실상 위키에서 검증된 패턴의 일반화다.
2. **위키를 고칠 것은 거의 없다.** 필요한 것은 연결선 3개 — 피드백→confidence(Phase 2),
   환류 생성 유스케이스(Phase 4), 평가 게이트 편입(Phase 5) — 이며 전부 예약석(`CONVERSATION` enum,
   `clamp_confidence`)이 마련돼 있다. §3 오버레이 그림의 점선 박스가 그 전부다.
3. **결정할 것은 하나다.** org 공유 지식의 수용처(위키 scope 확장 vs agent_memory 담당)를
   Phase 4 Design에서 확정한다. 기존 agent 스코프 동작은 어느 안에서도 기준선으로 보존한다.
